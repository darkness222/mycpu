# 傻瓜式教程：myCPU 答辩版

这份教程不是讲源码细节，而是帮你在答辩时清楚地回答：

- 这个项目到底是什么
- 怎么启动
- 每个按钮干什么
- 怎么证明它不是“假后端”
- 老师追问时怎么答

## 1. 一句话介绍项目

可以直接这样说：

> 这是一个带前后端联动的 RISC-V 模拟 CPU 系统。后端负责汇编、执行和维护 CPU 状态，前端负责可视化展示；主站演示和小恐龙游戏都共用同一个后端模拟器。

再短一点也可以：

> 我们做的是“能执行、能展示、能交互”的模拟 CPU，不是只画界面。

## 2. 启动步骤

### 最简单的方法

双击或执行：

```bat
start.bat
```

它会自动做这些事：

1. 编译后端
2. 启动后端服务
3. 启动前端页面
4. 打开浏览器

默认地址：

- 前端：`http://localhost:3000`
- 后端：`http://localhost:18080`

## 3. 主站怎么演示

推荐顺序直接照着走：

### 第一步：证明后端在线

看页面左上角的“后端已连接”。

你可以这样说：

> 这里前端会周期性请求后端状态，如果后端没起来，这里会直接显示未连接。

### 第二步：点一个普通示例

比如先点“默认程序”或“UART”。

要点：

- 普通示例点击后会自动加载并重新汇编
- 不需要再手动点一次“重新汇编”

你可以这样说：

> 我点示例时，前端会把汇编代码直接发给后端 `/assemble`，后端汇编后装入 Simulator。

### 第三步：解释“重新汇编”按钮

这一步老师容易问。

标准回答：

> 普通示例点击时已经自动重新汇编了；“重新汇编”按钮主要给自定义程序用。自定义程序允许我改文本，改完后再手动发给后端重新汇编。

### 第四步：单步演示

依次演示：

- `单步阶段`
- `单步指令`
- `连续运行`

重点看：

- 程序视图
- 执行轨迹
- 寄存器摘要
- 当前执行态

你可以这样说：

> 这些信息不是前端自己算的，而是后端每执行一步以后把状态返回给前端，前端只负责显示。

### 第五步：切到流水线模式

点击“流水线”。

看：

- 流水线时间轴
- Stall / Flush
- 分支预测失败

你可以这样说：

> 多周期模式适合讲单条指令怎么走五个阶段；流水线模式适合展示多条指令重叠、旁路、stall、flush。

## 4. 小恐龙怎么演示

点击“小恐龙游戏”。

你可以这样说：

> 这个游戏不是纯前端小游戏。后端会读取 `dino_game.s`，汇编并装入同一个 Simulator，然后每帧执行一部分指令，再从模拟内存里读出分数、障碍物和跳跃状态返回给前端。

再补一句：

> 前端只负责输入、传输和 Canvas 画面渲染。

## 5. 怎么证明不是“假后端”

这是答辩最关键的问题之一。

你可以从四点答：

### 5.1 主站必须请求后端

主站依赖这些接口：

- `/assemble`
- `/get_state`
- `/get_instructions`
- `/step`
- `/step_instruction`
- `/reset`

没有后端，这些按钮就不能真正执行。

### 5.2 前端不自己解释执行指令

前端只保存后端返回的状态：

- PC
- 寄存器
- 执行轨迹
- Trap / CSR / MMU
- 外设状态

它不在浏览器里再写一套 CPU 解释器。

### 5.3 小恐龙也依赖后端

游戏页走：

- `POST /game/init`
- `POST /game/step`
- `GET /game/get_state`
- `/ws`

后端还会从模拟内存 `0x2000` 一段取游戏状态。

### 5.4 WebSocket 不是摆设

小游戏优先走 `/ws`，如果 WebSocket 断开才回退 HTTP。

所以这里不只是“有个后端 API”，而是确实存在实时通信链路。

## 6. 老师问“你们各模块是不是分裂的”怎么答

可以直接这样说：

> 不是分裂的。我们把主站演示、游戏演示、多周期核、流水线核都统一挂在 Simulator 下面，内存、总线、设备状态也都走同一套后端对象。前端各面板也尽量从同一份状态快照渲染，避免每个面板自己算一套结果。

## 7. 指令集能力怎么回答

标准版本：

> 当前已经实现 RV32I 和 RV32M，并支持多周期与流水线两种执行模式，另外还有 Cache、分支预测、MMU、Trap、UART 的展示能力。RV32V 和 SIMD 目前没有实现。

如果老师追问矩阵乘法：

> 矩阵乘法 RV32I demo 是标量重复加法，RV32IM demo 是标量 mul，不是向量化实现。

## 8. 常见追问答案

### 问：切换示例是不是还要点“重新汇编”？

答：

> 普通示例不用，点击时已经自动重新汇编并加载了。只有自定义程序改完文本后需要手动点“重新汇编”。

### 问：为什么要做执行轨迹？

答：

> 为了让答辩时每一步能解释清楚。比如取指、译码、执行、访存、写回分别发生了什么，老师一眼就能看到。

### 问：为什么要同时做多周期和流水线？

答：

> 多周期适合教学和逐阶段讲解，流水线适合展示真正的重叠执行、stall、flush、旁路和分支预测。

### 问：你们为什么要做小游戏？

答：

> 因为小游戏能证明这个系统不是只会跑静态 demo，而是能持续执行、实时交互，并把后端状态转成前端可见结果。

## 9. 演示时最容易说错的地方

下面这些说法不要说错：

- 不要把矩阵乘法 demo 说成“向量运算”
- 不要说前端自己在浏览器里执行 CPU
- 不要说小恐龙是纯前端小游戏
- 不要说切换所有示例后都必须再点一次“重新汇编”

## 10. 最稳的一套收尾话术

最后可以这样总结：

> 这个项目的重点不是单独某个模块，而是把汇编器、CPU 核、流水线、多周期、内存、外设、RPC、前端可视化和小游戏演示整合成一个完整系统。主站和小恐龙都依赖真实后端状态，不是前端伪造结果。

## 11. 如果老师让你现场操作

最稳流程：

1. 先展示后端已连接
2. 点“默认程序”
3. 单步几次，看执行轨迹和寄存器
4. 切到“UART”或“地址越界”看设备 / Trap
5. 切到“流水线冒险”看时间轴
6. 再进小恐龙，证明是同一套后端体系

照这个顺序，基本最不容易乱。
