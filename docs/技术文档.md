# myCPU 技术文档

## 1. 项目目标

myCPU 是一个用于课程展示和答辩演示的 RISC-V 模拟 CPU 项目。项目重点不只是“能跑指令”，而是把以下能力串成一条完整链路：

- 汇编源码装载
- CPU 执行
- 流水线 / 多周期状态观测
- 总线、内存、设备、Trap、MMU 联动
- 前端实时展示
- 游戏演示

项目目标是“一个系统整体协作”，而不是各模块各做各的。

## 2. 总体架构

```text
React 前端
  ├─ App.jsx：主站演示与状态展示
  └─ DinoGame.jsx：小游戏渲染与输入
        │
        │ HTTP / WebSocket
        ▼
RpcServer
  ├─ /assemble /get_state /step /step_instruction /reset /set_mode
  ├─ /game/init /game/step /game/get_state
  └─ /ws
        │
        ▼
Simulator
  ├─ Multi-cycle CPU
  ├─ Pipelined CPU
  ├─ Memory
  ├─ Bus
  └─ Devices
```

关键设计：

- 前端不自己解释执行指令。
- 所有可视化信息都尽量来自后端统一状态快照。
- 小恐龙游戏与主站复用同一个后端模拟器体系。

## 3. 后端模块说明

### 3.1 Simulator

`Simulator` 位于 [RpcServer.cpp](/d:/Code/Project/mycpufinal/mycpu/src/rpc/RpcServer.cpp) / [RpcServer.h](/d:/Code/Project/mycpufinal/mycpu/src/rpc/RpcServer.h) 中，是前端和 CPU 核之间的统一入口。

负责：

- 维护当前模式：多周期 / 流水线
- 管理当前 CPU 核对象
- 装载汇编程序 / 二进制 / ELF
- 对外提供 `step()`、`stepInstruction()`、`reset()`、`getState()`

这意味着前端永远不直接操作 `CPU` 或 `PipelinedCPU`，而是统一通过 `Simulator`。

### 3.2 CPU 与 PipelinedCPU

位于：

- [CPU.cpp](/d:/Code/Project/mycpufinal/mycpu/src/cpu/CPU.cpp)
- [PipelinedCPU.cpp](/d:/Code/Project/mycpufinal/mycpu/src/cpu/PipelinedCPU.cpp)

职责：

- 指令取指、译码、执行、访存、写回
- 更新 PC、寄存器、统计值
- 维护执行轨迹
- 维护当前阶段和阶段视图
- 与内存、总线、设备、Trap 处理协作

当前支持：

- `RV32I`
- `RV32M`
- CSR 指令的一部分
- Trap / Interrupt
- MMU / 页表展示

未支持：

- `RV32V`
- 向量寄存器
- SIMD 执行通路

### 3.3 Decoder / Assembler

位于：

- [Decoder.cpp](/d:/Code/Project/mycpufinal/mycpu/src/cpu/Decoder.cpp)
- [Assembler.cpp](/d:/Code/Project/mycpufinal/mycpu/src/assembler/Assembler.cpp)

职责：

- `Assembler`：把文本汇编转成机器码
- `Decoder`：把机器码转成内部 `Instruction` 与反汇编文本

主站示例程序和游戏程序集都要先经过汇编器，然后再交给 `Simulator` 装载。

### 3.4 Memory / Bus / Devices

位于：

- [Memory.cpp](/d:/Code/Project/mycpufinal/mycpu/src/memory/Memory.cpp)
- [Bus.cpp](/d:/Code/Project/mycpufinal/mycpu/src/bus/Bus.cpp)
- [Device.cpp](/d:/Code/Project/mycpufinal/mycpu/src/devices/Device.cpp)

职责：

- `Memory`：存放程序和数据，支持分页映射
- `Bus`：连接 CPU 与 MMIO 设备
- `Devices`：提供 UART、Timer 等外设行为

前端看到的 UART 输出、Timer 值、外设中断，并不是前端本地模拟，而是后端状态回传。

## 4. 主站链路

主站前端在 [App.jsx](/d:/Code/Project/mycpufinal/mycpu/web/src/App.jsx) 中通过以下调用驱动后端：

- `assemble(source)` -> `POST /assemble`
- `refresh()` -> `GET /get_state` + `GET /get_instructions`
- `stepStage()` -> `POST /step`
- `stepInstruction()` -> `POST /step_instruction`
- `reset()` -> `POST /reset`
- `switchMode(mode)` -> `POST /set_mode`

链路如下：

1. 选择示例程序
2. 前端向 `/assemble` 发送汇编文本
3. `RpcServer` 调用 `Assembler`
4. 汇编结果交给 `Simulator::loadProgram()`
5. `Simulator` 把指令装入当前 CPU 核
6. 前端再通过 `/get_state` 和 `/get_instructions` 刷新程序视图与状态

注意：

- 普通示例点击时会自动完成“加载 + 重新汇编”
- 自定义程序不会自动汇编，必须手动点“重新汇编”

## 5. 小恐龙游戏链路

小游戏前端位于 [DinoGame.jsx](/d:/Code/Project/mycpufinal/mycpu/web/src/DinoGame.jsx)。

### 5.1 初始化

1. 前端优先尝试连接 `/ws`
2. 若失败则退回 HTTP
3. 请求 `/game/init`
4. 后端读取 `game/dino_game.s`
5. 汇编后装入 `Simulator`

### 5.2 每帧执行

每帧前端发送：

- HTTP：`POST /game/step`
- WebSocket：`{"type":"game_step","jump":true|false}`

后端会：

1. 根据输入写入游戏控制状态
2. 调用 `simulator_->stepInstruction()`
3. 从模拟内存 `0x2000` 一段读取游戏状态
4. 返回给前端

因此：

- 跳跃、分数、障碍物不是前端本地计算
- 前端只负责输入事件、通信、Canvas 绘制

## 6. 前端展示如何保持同步

前端所有主要展示都来自统一状态对象：

- `pc`
- `registers`
- `trace`
- `stats`
- `stageViews`
- `pipelineLatches`
- `peripherals`
- `csr`
- `mmu`

主站将后端 JSON 统一转换为 `sim` 状态，再驱动：

- 程序视图
- 当前执行态
- 流水线时间轴
- 执行轨迹
- 寄存器摘要
- Trap / CSR / MMU / UART 面板

这样做的目的是避免“每个面板自己算一套状态”。

## 7. 指令能力说明

### 已实现

- RV32I 常用整数指令
- RV32M：
  - `mul`
  - `mulh`
  - `mulhsu`
  - `mulhu`
  - `div`
  - `divu`
  - `rem`
  - `remu`

### 未实现

- `RV32V`
- 向量寄存器文件
- 向量访存
- SIMD 并行执行通路

### 关于矩阵乘法 demo

- `矩阵乘法 RV32I`：标量重复加法
- `矩阵乘法 RV32IM`：标量 `mul`

这两个 demo 只是用来对比“没有乘法指令”和“有乘法指令”时的执行成本，不代表向量化。

## 8. 执行轨迹设计

执行轨迹现在的目标不是完整调试器，而是“答辩时能看懂每步做了什么”。

设计原则：

- 后端负责生成原始 trace
- 前端负责把 trace 格式化成稳定中文描述
- 前端只显示最近若干条，避免整页铺开

这样即使后端某些 trace 历史上是英文，也能在前端统一成演示友好的中文。

## 9. 对“假后端”的约束

本项目明确禁止以下做法：

- 前端自己维护一套独立 CPU 执行结果
- 前端自己伪造寄存器、PC、执行轨迹
- 小恐龙前端自己本地跑一套游戏物理和分数逻辑，再伪装成后端结果

允许的只是：

- 前端对后端返回数据做格式化和可视化
- 前端对 trace、mode note 等文本做本地中文化整理

## 10. 已知边界

- 主站主要使用 HTTP 交互，不是持续流式推送。
- 小恐龙的“实时通信”优先使用 WebSocket，但会兼容回退到 HTTP。
- 当前 UI 更偏答辩展示，不是专业级调试器。

## 11. 结论

myCPU 不是“一个 CPU 核 + 一个独立前端 + 一个独立小游戏”的松散拼接，而是：

- 一个统一的 `Simulator`
- 两种 CPU 运行模式
- 一条共享的总线、内存、设备链路
- 一个主站展示入口
- 一个小游戏展示入口

它们共同组成一个“可执行、可观察、可讲解”的完整模拟 CPU 系统。
